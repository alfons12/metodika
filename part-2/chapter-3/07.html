Screen

Screen and Screen fragment artifacts describe contents and behaviour of user interface elements such as windows, dialogs or web pages. They don’t need to represent final layout, colors and other visual attributes, the important thing is to verify the content and behaviour of the screen with the stakeholders within specification phase or to quickly learn the goal and basic features of the screen after the release of the system.

Is it worth spending time describing user interface?

The screen artifact is basically a model so it has the same benefits as creating any other model. Analyst or any other team member doesn’t need to go to the source code when the it is needed to find out what exactly the Save button does. So it is very important to have screen documentation when the system is already in production.

There could be a complicated business logic or integrations behind each button or data table and it is not sufficient to document just the fact that “The button saves the customer data.” If you’re asked by your manager or a stakeholder you need to find the information fast and you mostly can’t afford to reverse engineer tons of code even if you understand the code. Another reason is that it is not always simple to simulate some scenario. You need to prepare data, ensure all integrations work just to see how the list of items looks. It is easier to check documentation.

Also, documentation the screens become the information hubs the same way as the use cases are during specification.

It is also equally important during analysis. Screens also become part of requirements. In early phases analysts usually elicit the basic use cases and requirements which are gathered in text form. They may also include user interface requirements such as “The product table should include Name and Price columns“ related to the use case ”Update order”. But it is impossible to gather all UI requirements in text form, moreover it is an antipattern. Visualizations support stakeholder’s imagination and helps design the solution faster.

Keith Pelletier writes on a LinkedIn discussion, "One big advantage is that it is so much more intuitive to discuss the fine points of software using visual prototypes. When dealing with abstract requirements (words), there is a lot of room for visual interpretation. In a sense, everyone is forming pictures in their head anyway. Why not just make it objective by creating a common picture that everyone can point at and discuss."

The most beneficial they will be for developers. Developers don’t code use cases (TBD uvod do artefaktu), the develop artifacts. They need to know what elements will be on the screen, how should the elements should behave, what data the screen should display or send out and where it should data get from or send to.

Screen layouts don’t replace written user and functional requirements. - p. 189, wiegers TBD

On the other hand, analysis can’t consist just from the screens. It is very hard to get the overall idea from the single screens. There must by a glue showing how the screens are related. Use cases of screen flows can be such glue. If the system should have a user interface, the system specification always consists from both written requirements and screen descriptions. This implies that the screen descriptions become part of the contract between you and the stakeholder about how the system should work.
