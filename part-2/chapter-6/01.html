Documentation in different environments

Although the documentation of common "business" systems should consists of standardized artifacts, techniques and tools so that it is easy to share the knowledge without learning different documentation styles the way the documentation is done is also dependent on the environment you work in.

From the project point of view

First aspect which might influence your approach to documentation is the type of the project.

Developing new system - green field project

Developing a new product is from the documentation point of view the most straightforward case. If there's no prescribed rules of specification issued by the customer the development team is basically free to choose how the documentation will look. This applies both to the system specification and documentation of deployed system.

The team creates the most appropriate artifacts which are most beneficial for the given type of project and which specifies the system best for the implementation. Example of these artifacts are textual requirements, use cases, business rules and various diagrams.

However, despite developing the system from scratch it is more than likely that it won't operate in isolation and it will integrate with other systems. It will either use external interfaces or expose its own intrface to the world.

To keep the documentation of all interfaces consistent the interfaces consumed by the systems need to be described the same way as the internal interfaces. For example if the system being developed calls a service to get the tickets from Paris to New York, the specification must be obtained from the airlines and transformed to the standard interface artifact so it could be used in specification and documentation

Moreover, these external interfaces might be consumed by multiple products so their specification can't be part of any of these products and should be shared among them.

Developing new system on platform

A lot of software companies don't do custom development from scratch but rather build their solutions on existing platforms. This includes customized versions of popular ERP or CRM systems or building systems on platforms such as Sharepoint or JIRA for instance.

Regarding requirements in the early stages of the project it doesn't matter if some platform will be used or not because the requirements are independent on the technology.

However, analytical artifacts are mixed from those provided by the platform and will just be customize and those which will be created from scratch. For exapmle if there is a Contact detail screen in the CRM system which is going to be deployed to the customer and two custom fields will be added to it, the there needs to be a artifact "Contact detail" created in the system specification to mamke it clear what fields in which particular screen the change should be done. The important message is that you need to partly document the platform functions. But only those parts for which you change its standard behaviour which you reference in your documentation as the standard platform behaviour should be desribed in its documentation provided by the vendor.

Custom software multiple deployments

A specific case of a "platform" can be your company's own product.

Custom developed software for a customer was a typical green field project. After it was deployed another company approached you with the request for the same software. Of course the implementation will never be 1:1 so the solution needs to be customized more or less. Your product becomes a platform and the documentation approach is exactly the same as if it was external platform. The scope of the documentation is bigger though. Since you company is the owner of the product/platform you need to document it as a whole.

Modifying existing system - brown field

When adding new features or altering existing ones it depends whether the system was developed by the company implementing the change or the system was developed by a third party.

If the company is modifying system which it developed it can use its own documentation. If it's a third party system the main features need to be mapped and at least artefacts related to the changed parts need to be created.

In general as far as the documentation is concerned there shouldn't be any difference between the system developed intrnally or a third party system in terms of style, extent and quality of the documentation. If the company wants to build new features to the system it needs the documentation regardless of who is created the system. The documentation can be created in cooperation with the author of the system but it needs to be created anyway.

From the organization type point of view

Appart from the project type a kind of organization operating the systems influences the of work with documentation.

Software house

Software development companies aka software houses implement custom developed IT solutions for various entities. These solutions could new systems built from scratch, using a platform, they can modify existing solutions both theirs own or taken over from another vendor or they can just integrate existing solutions.

For big software companies this means keeping documentation of dozens of systems for different customers each having multiple versions plus documentation of external system and its interfaces which the internally developed communicates with.

Developing a software for customer can from the documentation point of view mean a documentation duplication as both sides, the vendor and the customer, need to know how the system works and tend to create their own version of documentation.

Internal IT department of an organization

Large banks, insurance companies, pharmaceutical companies or engineering companies they all have their own IT departments responsible for operating their systems or even custom development. Such companies operates dozens of systems for various vendors. They are in role of both customer and vendor so it is very common for them implementing projects which impacts systems developed internally and from two or three vendors at the same time.

For analysts this is quite challenging because analysis involves communication with different entities. Not having a single standardized documentation across the systems therefore implies analyzing tons of documents, asking dozens of people which turns analyzing activities to detective activities which is both frustrating and time-consuming. Also creating specifications is hard if the artifacts for all systems are not desribed and could not be reused in the specifications.

These institutions thus need to manage documentation of their own internally developed systems, systems delivered by the vendor, systems developed using some platform, external systems exposing interfaces etc. There should be a single place listing all applications and their integrations. Each system shoudl have a overview describing its goals, basic functionalities, intergrations etc.

On top of that, organizations of this size operates in many domains so there should be a solid domain model desrbing the business, products, basic processes, operations etc. Otherwise it is very hard for analysts who are not dedicated to one problem subdomain to switch when new project from different domain starts.

Governors vs. invaders

There are also two different types of people writing documentation which has a great influence on the quality of documentation.

First are people who maintain the legacy systems - the governors. They work with the system and its documentation on the daily basis and tune it to almost perfection because it has an impact on their day to day performance and comfort. They govern the documentation protecting it from any intruders who might do something wrong with it.

The downside is that these governors know the system from A to Z and they don't care how the information is documented. It is only important for them that it is documented and since they know the structure and rules of the documentation by heart they don't have troubles find it anywhere in it no matter the form.

This sreriously impacts newcomers who try to take over the documentation and they feel lost. Governors don't understand how could be possible that the new people can't keep up and work effectively if they can.

Governors should be trained how to structure the documentation, how to model information systems and they should try to standardize the documentation with governors of other systems so that it is easy to change the camp and start governing another camp very quickly.

The opposite to governors are invaders. They don't care about campsm or villages, they were ordered to invade a teritory and build up a wooden cottage house to test whether it is possible to live in the teritory at all. Deadline in 3 weeks, no plans needed.

Invaders are teams asked to build a concept or a prototype but quickly. The problem is that invaders conquere the land and leave. The deadline and features have the priority, agile development is an alibi for not documenting anything and in the end the company ends up having a undocumented prototype which survives next couple of years in production.

The advice here is to plan at least little time for documenting at least high-level information, architecture and decisions so that it is put on paper when it is fresh making the governors' live easier when they take over the teritory.
