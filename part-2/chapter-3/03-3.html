Good practices

Learning the purpose of use cases, its components and to write a simple scenario is easy and analysts adopt it very quickly. Learning to write consistent use cases, which are all at user-goal level and use consistent language is harder. Mastering it requires training and experience and the more use cases analyst writes the better they are.

But there are also some good practices which can't guarantee the success but at least support it.

Use case components

From the reader's perspective it must immediately clear what is the scope of the modeled use cases and who benefits from the use cases. This is effectively done by indicating the system under discussion and the primary actors.

System boundary

System boundary represents the functionality to be implemented and thus indicating the scope.

Always include the boundary to make clear what system we're talking about. Use a meaningful name for the system boundary, preferably just reference a system object not to duplicate the information.

Actor

  1. Name Actors With Singular, Business-Relevant Nouns
  2. Actors Model Roles, Not Positions or real persons
  3. Actors Don’t Interact With One Another
  4. Actor musi mit popis podrobnej nebo odkaz ,actor is just a link to unique actor

System and Time

Sometimes there is a need to describe a functionality which is not triggered by a user by an event. The common practice supported by many influencial people in the field of software engineering is to model it as use cases performed by a System actor or a Time actor. For example use case "Delete all unactive customers" is triggered by Time each 6 hours.

From the UML perspective it is nothing wrong about that and developers will understand what you mean. But what about the fact that no actor doesn't benefit from the use case which is the main assumption? Well, we're not academics, we need to build the systems fast and despite this inperfecton is seams that the concept works in general.

Just be careful to stay at user-goal level and avoid implementation details when desribing use cases triggered by time. As there is no user-system ping pong it is tempting to inlcude more detail than it should be.

UC Delete all inactive customers

  1. System goes through all customers not marked as inactive
  2. If the customer hasn't ordered any item in the last 12 months
    • System marks the customer inactive
    • System sends an email to the customer wihh a promotional code
  3. Otherwise system proceeds to the next customer
  4. Use case ends

It is clear from the example that it is tempting to include more implementation details when describing use case triggered by time event. It is natural as there is no user-system ping pong so analyst unintentionaly switches context and includes these details.

Another requirement can be to send an email to a customer everytime customer's address changes. Who is the actor there? It is neither a user nor a time. The actor is an event which occured in the system.

If the address could be changed only by one system and one use case, this requirement would become an activity inside this use case. Unfortunatelly, there is no single primary function which changes customer's address. To avoid any tecnical decisions at this level, we would recommend to model this requirement a using Time event which periodically detects the customers whose address has changed and reacts accordingally. The technical solution will be decided afterwards.s

Tips for writing effective use cases

Do not tune them to perfection

just good enough - odkaz na kocept

Use active verbs

Write the goal as an active goal phrase: <verb> <direct object>

For example Book a ticket, Issue an ivoice, Open an account.

It helps analysts think from end user’s perspective. If you name use case Booking a ticket or Books a ticket it is still understandable nevertheless it should be consistent among all analyst and the defacto standard is using the active verb.

If you name it "Issued invoice" or "Yearly report", readers can only guess what the analyst meant to do with the invoice or report. Without the verb it is impossible to tell the user goal.

Short scenarios

Use case scenario should include max 5 to 9 steps. Only this way it is easy to read and most beneficial. If the scenario is longer, agregate the steps and break them down in refrenced activity TBD link

Terminate the flow

It is a good practice to end the use case scenario with the statement like "The use case ends" or something like this to indicate that the use case was finished to end by analyst. Only like this can everybody be sure that the use case is not still under development.

Use case packages

Big enterprise systems can consists of dozens of use cases. Obviously, it is not a convenient strategy to have all of them in a flat structure so the good practice is to put related use cases to packages.

For example all use cases related to invoices would be placed in the "Invoice management" package and all use cases dealing with customers would go to "Customers" package.

Similar to 5 to 9 rule applied to the scenario length it applies also for use case packages. Put 5 to 9 use cases per package which will enable you to show all of them consistently in one use case diagram without making it too complex.
