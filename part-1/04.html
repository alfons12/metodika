Frustrated analyst

We have worked on many various projects in many different roles including developers, analysts, testers etc.

While it is natural that starting working on any project requires going through supporting materials and accustom to them we quite often struggeled to read the matterials effectively at all.

Many times, we were presented with a bunch of unrelated documents, diagrams and other materials which instead of helping to orient ourselfs in the domain, process or software leeaving us desperate.

You may ask how the legacy member of the team worked with the supporting materials. Did they use it at all.

Yes they did because even when there is a junior team or a team with no deeper knowledge of analysis (trends etc TBD) the team just believe that the approach to analysis and documentation which was presented to them in the beginning is the right one and get used to it.

Another aspect in this is time. After spending a couple of months even years on the project or product people simply learn the domain, the processes and the whole application. The poor documentation then becomes a help only in case of complicated processes or algorithms. They are helpful to the team even when the information is buried in the document called FS20170526-A.23 and desribed only in text taking 2 A4 pages.

And of course as the old analysis which now became a documentation also the new analysis, change requests or project charters are produced in the same quality.

However, for people like us who often hold a role of a consultants moving from one project to another this represents a inexhaustible source of frustration.

Sources of frustration

Poor documentation

Note: By documentation we mean all artifacts which are produced during or after analysis such as overviews, models, pseudo codes etc. whose purpose is to abstract from low-level details and provide simplified view of the problem to help understand it. We don't mean technical documentation describing source code artifacts like classes, methods or procedures.

Have you ever seen a software documentation which is not poor? We neither. Does it mean we are condemned to frustration forever? Definitely not.

To choose the proper way of handling documentation problems the cause must be identified first.

  1. Missing documentation Solution: Start document software, processes and other artifacts.
  2. Inconsistend documentation Solution: Introduce at least basic rules to follow when documenting artifacts, train analyst so that they know about them and do quick reviews.
  3. Documentation is vague or too general Solution: TBD

We will tackle all three scenarios in the next part (TBD).

Reasons to have documentation

We are not fans of documenting. Nobody is. So why should we do such unpleasant activity at the first place?

  1. Knowledge tranfer - it must be easy for new members of the team or anybody to orientate in adn to undersand the problematics (process, system)
  2. System features - it must be easy to find out how the system works without digging into source code

  3. Architecture - It must be easy to automatically see the architecture, integrations and dependencies of the systems

  4. System documentation - it mmust be easy to orientate in the sysem fast (what are the main modules, screens, how the system integrates with other systems)
  5. In general - it must be easy to

Missing documentation

Let's elaborate a bit why so many teams create very poor documentation or even no documentation.

  1. It is a drag to create it Finding the place where the documentation should be takes a long time. It is poorly written, it is not clear how to document the information, there are no rules and it seems that nobody cares at all.

    We believe that we are able to demolish most barriers with Effective analysis.

    We believe that proper specifications and documentations are not written not because of analysts are lazy but because nobody helps them to create it fast, intuitively and effectively and nobody requries them to do so. The quality of software corresponds with that fact.

  2. Source code only It is because of the misunderstanding what agile community says that documentation is not needed. People often disinterpret agile manifesto in the way that there's no need for the documentation because source code is enough.

    Don't get us wrong, we are fan of agile and we try to do our projects the agile way. What's wrong with agile is that in hands of people who didn't get the point of agile it might be dangerous for everyone.

    What agile say, is "Working software over comprehensive documentation". Over, not instead.

    Everytime we come across a team saying that they don't need documetation because source code is the documentation, we start to smile.

Source code only - why not

We already talked about generalized specialists. Those people know how to proceed with analysis and are able to read source code at the same time (simplified). In teams consisted only of these people it should be ok to have just soruce right as everybody understands it, right?

Unfortunatelly not. With the complexity of current applications and systems it is simply not possible to be effective without any simplification and abstraction. Teams simply need model and overviews to abstract from low-level details.

  1. It is certainly faster to locate a model or a function overview to find out information than to search through the source code. Whatsmore, source code can be written very bad, it can't give you high-level view etc.
  2. Source code says how the application works not how it is supposed to work "That's your problem, it should've been tested...". Yes, it shoul've been, we're not living in the ideal world though.
  3. Source code documents only the system it represents. Documentation descibing how the system are integrated, how the communication flows and what are the dependecies can't be read out of source code in reasonable time.

Too general documentation

Story we hear very often is that the documentation is only high-level because this way it stays up to date even when some detail changes.

Have you ever heard saying "The devil lies in detail."? You don't want to mess with the devil.

Verbal specification

Another frequent story we hear is that documentation was not create because developers know the system so well that it is just enough to tell them what to implement and they'll do it.

Well, no problem there as long as at least a rough description of the functionality is put somewhere so that it could be tracked who required such functionality and why it even exists.

What is a problem though is mixing the specification for development wih the documentation. Basicaly, if the software process where the instructions for developers are provided only verbally works, then ok. However, it will strike you back once the half of the team needs to be replaced.

Duplications, inconsistencies

Information sprinkled throughout various documents

Use case description in Word document, decision table required by the use case in the linked Excel sheet, use case realization in activity diagram drawn in CASE tool.

Impossible to search using one single query and uncomfortable to navigate between artifacts.

It is not possible to find information with one single search, it is needed to join various sources

Information can be found on many different places

Even worse is that the information found on many spots is also often contradictory, e.g. two versions of algorithm description, system is sometimes called CRM sometimes only CM.

Each software artifact desription or model looks differently

There is no rule how business need description, use case or screen specification should look. It is then harder to read the documentation as one needs to search for the particular information instead always going to the same place. Since there are no rules each analyst invets hisown which makes the specifications and documentation total mess.

Incoherence

By incoherence we mean that all related information can be found at the single place. The location of information should be predictable.

Example:
We saw a screen specification which included a wireframe and a list of user interface elements like buttons, inputs etc. However, the detailed desription of each field (default values, validations etc.) were kept in separate Excel sheet.

It was so frustrating when investigating the wireframe to open separate sheet and search for the element everytime a single button functionality was needed.

Coherency here means that you get all you need to understand the screen in one specification. The same applies to ather artifacts like diagrams for example. When going through a process diagram you should not need to rebound to a separate material to read the details about each step. It should be coherently place together.

We also don't like if there is a standalone list of business rules. Coherence for us means that the things are put into context. Reading just business rules without the context where they are applied is useless. Coherency is if the rules are also attached to business process step or system function where it is applied.

Mixing abstractions

In the previous chapter we demonstrated how analyst go through different levels of detail starting with the process, ending with the data model.

This separation of detail is natural during the analysis phase as the ongoing analysis uncovers more and more detail of the problematic. This is actually the aim of the analysis to start with a high-level problem and to refine it to artifacts which are understandable, desriptable and implentable.

What is not natural for many analysts is to keep the levels of detail separated.

Mixing all information together ends up having jumbo specifications which are not either understandable nor maintainable.

Apt example of this common mistake is a "use case of mass destruction". It doesn't need bo exclusively a use case it can be simply any artifact whose description mix up information from different level of abstraction or details.

Example

In the previous example describing ordering plane tickets we also commented on the inappropriate decision to describe it all in the text. But it also represents a jumbo use case and clearly demonstrates problems associated with mixing different levels of details and different views on the problematics. It describes the process, it also describes user interface details and even low-level implementation details such as interface URLs and integration specifics.

   - System searches for United airlines tickets by calling a service at https://aa.com/tickets using GET method
   - The connection is secured using HTTPS protocol

What such specification doesn't give you:

  • Quick high-level overview of the process of buying a plane ticket online
  • Understandable description of what is going to be build which is readable even for the stakeholders so that they can comment on it or formally sign off if needed
  • Clear software artifacts which would serve as a specification for developers
  • Specification which is easy to maintain - iamgine you want to change checkbox Email delivery to a two-state button. You need to change it in wireframe and go through the whole jumbo specification and change it also here. Cruel.

Instead, it represents almost all previously desribed frustrations plus the new one "Text-only specification".

Such specification is cluttered and not easy to read for anyone. Textual form makes it huge, it is hard to find the information you need, readers are distracted by implenetation details, it uses inappropriate approaches, for example decision table could be used to determine the discount to make it easier to understand and so on.

Tell me Why!

Sometimes even the precise specification is no use for analysts if the goal of the task is to find out if the specific requirement, process branch or input field is needed. We ended up frustrated so many times when the “Why” came.

  • Manager approaches an analyst asking why it is not possible to pay for goods on the client’s website via Paypal anymore. There was a end-customer complaint registered and a new project manager on client’s side is desperate
  • It is a common feature of our software and it works for all other clients and nobody know why this particular client has the feature blocked.
  • This analyst is not responsible for the analysis of this particular client and unfortunately the main analyst responsible for analysis of the issues for this particular client left company month ago and there is still noreplacement.
  • Luckily a high-level description of the change was found in the shared folder. It describes in detail what stopping offering payments via Paypal means and what parts of the solution is affected.
  • However there is no mention who initiated the change and what was the business need behind it.

After 2 days filled with frustration on all sides - client PM, vendor PM and analyst, a contact on the owner of the request was found and on the scheduled meeting the reason was finally uncovered.

If only the company had a list of change requests describing the not only the functionalities but also the business needs and which led to implementing the functionalities.

Auditing

Let’s be realists, it is not often easy to find out the what is behind the requirement or software artifact. If the application requires user to enter his date of birth it is very rare that it is clear from any document why the application needs it and how it is processed once it is entered.

But analyst can always ask someone, right?. However, if it is also not possible to identify who is the author of the newly added paragraph, who was the main analyst of the change requests impacting my project then it is very frustrating when our work changes from analysis to puzzle playing.

If only the changes were audited…

Changing requirements

This is an inseparable part of the software development. People always tried to fight them, fixed them and locked them. After decades of fighting analysts found out that they can’t win. Instead, they take it as a natural thing and it became a one of the core principles of the agile manifesto:

Responding to change over following a plan

So it should not be about fixing requirements but rather about taking such approach and use such techniques and tools to be prepared for the changing requirements. Automatization, avoiding duplication and other approaches described in Effective analysis do a great job in handling the changing requirements.

Many people complain that they hate documentation because the stakeholder ideas, processes, systems etc. change so rapidly that it is hard to keep everything up to date and consistent.

We hear people crying very often over the documentation.

  • Document late
    • Document only things which are already deployed
    • Everything else don't consider documetation and don't insist on the same qiality as the documentation must have
  • Avoid duplication, single source information
    • If the information is in one place only it is easy to maintain the information
  • Don’t mix different level of detail
    • If the use case description is separated from screen description it's easier to identify the place where the information must be changed
  • Apply automation
    • If artifacts are generated automatically and they partialy correct themselves, documenting is less uncomfortable task
  • Have solid artifacts
    • TBD

Not 100% perfect documentation

We need to distinguish between missing documentation documentation which is impossible to read or modify, documentation which includes things which are crucial and are not correct anymore from the documentation which is consistent, easy to follow and useful but doesn’t include every detail.

I can tolerate a map that is missing one or two streets but I can't tolerate one that is missing three quarters of the streets in my town.

-- Scott Ambler

It is obviously not a problem if the button is red instead of blue or if it is not explicitly said that the default sorting of the table is by the Client name ascending. You simply run the app and see it.

What is problem though is that you have no idea what the button does, why it is there and from what external system it gets data. You can’t reverse engineer every application, you can’t spend hours looking to the source code or ask developers to find out every time you need it.

Prevent frustration

First we should all try to not frustrate other team members, current or anyone joining the team in the future.

This means to produce consistent, easy to read artifacts which comply with common rules across all products and projects.

What is that hard on producing high quality specifications and documentations and not spending ages on them? It is just about a couple of rules, educating team and introducing the right tools which make the process effective and automates the common task. It can be so easy and effective you wouldn't believe.
