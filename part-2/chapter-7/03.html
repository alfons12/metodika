Sequence diagram

For many analysts, sequence diagrams pose a kind of mystery. We don’t know why but sequence diagrams are not used as often as they would deserve. It might be partly because they look complicated on first sight partly because of poor support in “drawing” tools (not modeling tools) TBD odkaz a mozna udelat sekci drawing vs. modeling

Sequence diagrams are very useful for modeling interaction between entities (systems, objects, even business units) when the order of the interactions is important. In this scenarios they are irreplaceable helpers and our experience is it remains valid also for using them as a communication tool when discussing them with the business people. It is not true that they are not readable for business. Our experience proves otherwise as long as they are kept simple using only the basics from the notation.

Sequence diagrams depict calling methods on objects. It means that the arrow from object A to object B represents object A calls an operation object B. Object B needs to implement the method which is written on top of the arrow.

Apart from modeling the low-level objects interactions sequence diagrams are very useful for modeling high-level communication as well. This could include modeling a process, use case scenario or calling services between systems. This stems from the need to describe the high-level concepts to stakeholder using the tools analysts are familiar with.

In these cases the messages are not real methods but rather process/use case steps, service callings, data transfers or abstract capabilities of the entities - the screen allows user to enter product data for example.

The user doesn't call "Enter product data" operation on the screen, instead it sends a signal that wants to sign in. Similarly, the Central product system doesn't call Data warehouse's operation but rather sends data to it. In these cases the callers don't wait for the response.
