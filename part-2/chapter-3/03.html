Use case

In the previous chapter we talked a lot about requirements and especcially emphasized that they can be of many different levels of abstraction and can be described using many techniques from process diagram through a simple free text sentence to a complex algorithm described by a formal code.

Therefore, stakeholders usually end up having a big bag of mixed requirements which are hard to understand. So analyst needs to analyze the requirements, create separat smaller bags and place them into it so that a requirements hierarchy is build which helps everybody to understand them faster.

EXAMPLE

We need a software for managing invoices

  • Accountants must be able to create an invoice
    • System must automatically update grand total as the accountant adds invoice items
    • Accountant needs to select a customer from the registry of customers
      • System must provide an autocomplete function when entering a customer
  • System must be able to export the invoice to PDF

However, when focusing purely on software requirements even such separation of requirements is not enough. For example a requirement "User will be able to evaluate validity of an alert raised on contract" is not an atomic step. It consists of many steps which together make an interaction between the user and the system. Such intraction captures a contract for the behaviour of the system which is crucial for understanding how the user intends to use the system. It describes what user wants to do in the system and the basic steps which are involved. The capability of the system together with the usage scenarion is called a use case.

A use case is a list of actions or event steps typically defining the interactions between a role (known in the Unified Modeling Language as an actor) and a system to achieve a goal.
They describe the step by step process a user goes through to complete that goal using a software system.

EXAMPLE

UC Evaluate validity of an alert on a contract

Desribes how the back office employee evaluates whether the automatically raised alert is positive or not based on analysis of the conditions in the contract

  1. User choose an alert from the list of alerts in a queue and opens it
  2. System displays detail information of the alert and the unmet contract conditions which caused raising the alert
  3. User goes through the conditions one by one, opens attached document associated with the condition, compares the document to the respective part of the contract and evaluates whether the condition is met or not and marks it accordingaly
  4. User saves the changes
  5. If ale conditions were met system deletes the alert and marks it ready for approval. If not the contract is assigned back to the front-desk employee which needs to solve the problem with the client.

Use cases are becoming more mainstream as a method for capturing requirements, as evidenced by the endorsement of big companies and methodologists such as Booch, Rumbaugh, and Coad. One benefit of use cases is that each one encapsulates a set of requirements. This encapsulation lets you easily manage and track the use cases individually and provides a better alternative to prose requirements.

Note: We don't aim to provide a comprehensive material covering everything about use cases. There are a lot of books and other material available, here we just want to emphesize what is important from the Effective analysis perspective.

Use cases purpose

First, let's ask what is a use case? Is it a technique? Is it a type of requirement? Or even a requirement package? Well, it's all of it.

Typical use case scenario consists of number of steps where each of the steps are requirements themselves. Looking at the example above System displays detail information of the alert is a use case step and also a requirement what the system must be capable of. So a great benefit of use cases is they encapsulate a set of related requirements together which makes them great for organizing requirements coherently.

Next, use case itself is also a requirement as it also represents what the system enables the user to do - in this case it enables user to evaluate if the automatically created alert is legitimate or not. Use cases shouldn't be duplicated by or converted to other forms of requriements, only supplemented by them.

Finaly, gathering and analysisng use cases, called use case analysis, is considered a mainstream method for capturing requirements. It helps describe what the user needs to be able to accomplish when using the system.

Use case components

For each use case it is important to be clear what it does (Name and Description), who is the user performing the use case (Actor), and with which system the actor interacts (System/Boundary).

The core of the use case is obviously the scenario. To visualize use cases UML inlcudes a use case diagram. We'd like to emphasize that it is not a required part of you analysis. The only benefit is really just to visualize them so that it is clear what use cases are in scope of the system and which actors can perform particular use cases.

Where it fits

After creating a couple of use cases you quickly realize that it sometimes really tricky to distinguish what is a use case, what is its step etc. In other words, use cases share the same attribute with other types of requirements which is it can be described on many different levels of abstraction.

Use cases can be top level use cases such as Start a mortgage deal which represents a business goal or a long-running process, user goals such as Create an invoice which is actually what the user of the system wants to do with the system, or a low-level subfunction such as "Add customer to the invoice".

Use cases are neither business processes nor system functions

Besides the level of detail aspect, many people also tell what is a use case based on it value to the user. There is an unwritten law that a requirement or a functionality is a use case only if it's worth for the user to turn on the computer to complete it.

This is why there are so many questions regarding what is a use case or what is not. The bad news is there is no rule or guidance hwo to recognize it. However, there are a couple of handy indicators:

  1. Actor starting a use case is the only actor performing tasks within the use case (so it is not a process)
  2. Actor starting a use case interacts only with the one system under which the use case belongs. The system can call other systems but this is not visible to the user (use case is a capability provided to user by one system)
  3. Actor can finish the use case in one row in 2-20 minutes (this eliminates administrative/service use cases and long running processes)
  4. Actor benefits from the use case even if it's performed many times in a row (this elliminates use cases not bringing much value to user)

Looking at the use cases from this perspective use cases such us Sign in, Sort items by name, Add note to a contract or Search customers.

User goal level

The user goal level use cases effectively desribes the main behaviour the stakeholder expects from the system yet it still desribes enough detail for analysts break them down into more detailed artifacts and flows.

Use cases on higher levels than the user goal level provide values as well but the steps are too abstract for analyst to understand what the desired functions are. There is also a risk such use cases are processes including multiple systems which is not what analyst needs.

The system subfunctions are also neccessary in order to have them developed, however they can't be placed on the same level as user goal use cases because for complex systems you would end up having houndreds of use cases which would make your specifications impossible to read and review.

The common characteristics of the system subfunctions:

  • Very simple - they usually can't be desribed using a scenario (Mark email read, Filter contact list)
  • They are not invoked by themselves but they are mostly part of other use case's scenario (Find customer)
  • They include implementation details or just calls a function or a service ("Validate address")

Use case is not every function or every user-system interaction desribed by scenario

Requirements always change, if youve got good use cases though, you can usally change your sw quickly to adjust to those new requirements

Subfunctions

So how should the subfunctions be specified if they "are not use cases"? Subfunctions don't exist on their own so thei are always invoked from some use case or another subfunction. This answers where to put them.

Another question is whether to call them and model them as use cases. From the Effective analysis perspective there is only one kind of use case - user-goal. Everything else is just a step of the use case - an activity TBD link. The reason for this decision is that the practice has proved that including, parenting or allowing use case hierarchy always led to analysts having done functional decomposition. This should be avoided - TBD link avoid func decomp. Analysts then end up having dozens of use cases or using use cases as process steps which Effective analysis tries to eliminate.

On the other hand, at activity level the function decomposition doesn't mean a problem. Activities can shared by use cases, reused, broken down to details etc.

Secondary use cases

It shouldn't be hard to recognize the user goal use cases if you use the basic rules listed in this section. You quickly learn that requirements such as Filter contact list or Mark email read are not good candidates to be included in user level use cases.

However, despite you find the right level of abstraction and identify all use cases of the system you can't be sure that somebody will tell you that half of them actually are not use cases. This is because not all user goal use cases provides the same benefit to users. We call these use cases secondary use cases (service use cases or administrative use cases).

The common characteristics of the secondary use cases are:

  • They are invoked rarely - they are not the main system functions which user would use every day (Reset forgotten password)
  • If you removed them from the system it would remain functional yet limited (Export list of invoices to CSV)
  • They are simple CRUD (Create, Read, Update, Delete) functions without any scenario (Create product, Update product, ...)
  • They are service/administrative functions such as Update settings or Cancel long-running job

Assessing the business value of the use cases is really a matter of taste and each use case needs to be assessed individually.

So how should the secondary use cases be specified if they "are not use cases"? Well, if they are user goals, they are also use cases. It must be documented who can invoke them and what is their purpose. Just separate them from the core primary use cases and don't spend much time documenting them in detail.

Use cases and other requirements

Despite being beneficial method for capturing behaviour of the system, use cases themselves are not enough as they can't describe the system from all angles. They won't tell you how the screens will look, what are the core algortightms and calculations, how fast the system should be or what securit constraints need to be implemented. But, it can put all these facts together and serve as a hub which holds them coherently together. This help structure the requirements.

By putting together it is not meant that all the requirements should be included in the use case specification. They must be only reference for two reasons. First, the use case specification would become enourmosly big and second, the artifacts such as screen, business rules etc. are very likely to be shared among multiple use cases and inlcuding them would duplicate them.

TRAP

You analyze use cases for a transportation company system which handles the status of each package. So far, you've writte scearios for these use cases: Create package, Set package status, Change package status. You present it to the stakeholders and one of them says: "And we also need a report to see how many packages have not been delivered yet."

First confusion for analyst here is that stakeholders didn't tell analyst what he wants the system to do but instead what output the system shoudl produce. So analyst first needs to convert the requirement to a use case: "Genetate a report of not delivered packages".

Another trap for analyst is that the report is not in the scope of the system being analyzed. The system shouldn't be responsible for creating such report which migh not be clear unless the analyst si familiar with the overall IT architecture of the organization. He needs to recognize, that this use case is a part of the overall solution but it is not a part of the system that is supposed to be build.

So, you always need to convert user-goal to a form of use case and decide in which system scope it should be place.

Use case realization

Use cases scenarios should be detailed enough to describe how the user interacts with the system but they shouldn't uncover design decisions or implementation details such as which screens are opened, how the components of the system collaborate to fulfil use case's step etc.

However, as the analysis proceeds even these aspects of the system need to be described using use case realizations.

A Use-case realization describes how a particular use case is realized within the design model, in terms of collaborating objects.

Use case realizations define the design in terms of collaborating objects which support the use case. Therefore, each use case realization is typically made up of a number of diagrams, most commonly sequence diagrams or activity diagrams, showing the collaboration or interaction between the participating objects.

By its nature use cases are effective technique during the initial anlaysis phase where analyst needs to understand the overall picture of the desired behaviour of the modeled system. However, developers don't implement use cases. They implement screens, functions, interfaces, services and other more low-level artifacts. Similarly, the stakeholders can't think the required solution through on. They need to visual artifacts such as sreens, process diagrams or data models to realize the details and dependencies.

So, take use cases only as a first step to understand the scope and context but try to outline the design artifacts as soon as possible. Use cases will be thrown away or transformed to documentation overviews so don't spend much time tuning them.

Use cases and documentation

Since use cases are a form of requirements they will not be part of deployed solution documentation (TBD odkaz Requirements and documentation), at least not all of them and in the form they were written during analysis.

Use cases as the high level system functions can be beneficial as a basis for a system overview. However, the overview should outline only the most important features so definitely not all use cases should be included. Also the form of use cases used during analysis is not suitable for documentation. The description should less structured and should be put in more "documentational" style.
